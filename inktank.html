My Process

My design process for this project was a sticking to a strict method of research, ideation, problem solving, and sketching. Not every project is treated the same due to some being further along in the development cycle than others. Not everyone starts out the same way in the beginning. For InkTank, I was able to follow the following classic steps for traditional UX design.

Competitive Analysis

I tend to start a project by evaluating the playing field - scouting the competition and creating SWOT profiles to see what they’re doing in relation to the project’s goals.

Interviews and User Stories

Once I have some competitive analysis done, it’s time for brainstorming and user interviews. I want to get inside the head of users that would be using this product in order to find out their wants and needs so I can create a problem statement to guide the direction of the project. This is also a stage in which basic user stories are created in order to fulfill some features that the product may need.

Some entry level interviewing is ok at this point.

Personas and Journey Maps

Personas and journey maps can be extremely helpful for a large team or a project that spans months in development. They are great references to refer back to in order to keep the project on the same track in line with it’s original goals.

Sitemapes and Wireframes

The project starts to come together with a basic sitemap. This can tend to change slightly through the iterative process of design where new features may be added or revised. Wireframes start to give all stakeholders a visual layout of how everything will be organized.

Mid-Fidelity and High-Fidelity Prototypes

These prototypes help give real world users a feel for how the final product may look and behave. Not only is this helpful for marketing and product demos, it’s also a great opportunity to get user feedback on scenario tasks. Any input is recorded and placed into the iterative design process for additional improvements.

Conclusion

Like many UX designers, my process is somewhat identical. The reason being, is that it’s classic and proven. However in many cases, adaptability is easier than restarting a completely new design process - especially ones that have been in development for months. I strive to solve large problems within organizations. Often, it’s based on conversions and gathering leads whereby a design overhaul is needed.
